# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Common commands

- Bootstrap and run recipes without prior install: `uvx --from rust-just just <recipe>`.
- Install/sync the development environment and hooks: `uvx --from rust-just just init` (installs `rust-just` as a uv tool, runs `uv sync --all-extras`, installs pre-commit hooks). After `init`, recipes can be invoked as `just <recipe>`.
- Run the full lint/format cycle: `just lint` (Ruff check with fixes, Ruff format, then Ruff check again).
- Run all pre-commit hooks: `just lint-pre-commit`.
- Run the default test suite with coverage: `just test`.
- Run tests across the configured Python version range: `just test-all`.
- Run a single test file or test node: `uv run --extra dev pytest -v tests/test_templates.py` or `uv run --extra dev pytest -v tests/test_github_init.py::test_build_config_pulls_defaults_from_pyproject`.
- Build distributions: `just build` (`uv build`).
- Serve docs locally: `just docs`; build docs: `just docs-build`.
- Exercise the CLI locally: `uv run repo-scaffold list`, `uv run repo-scaffold create python --no-input --no-install -o <temp-dir>`, `uv run repo-scaffold create uv-workspace --no-input --no-install -o <temp-dir>`, `uv run repo-scaffold create react --no-input --no-install -o <temp-dir>`, `uv run repo-scaffold create rust --no-input --no-install -o <temp-dir>`, and `uv run repo-scaffold create vue-project --no-input --no-install -o <temp-dir>`.
- Exercise GitHub bootstrap without pushing: `GITHUB_TOKEN=<token> uv run repo-scaffold gh-init <generated-project> --no-input --no-push`.
- Exercise add-package in a generated workspace: `uv run repo-scaffold add-package <name> -p <generated-project>`.

## Project overview

This is a Python 3.12 package exposing the `repo-scaffold` console command. It is a Click CLI around bundled Cookiecutter templates plus a GitHub bootstrap helper for generated projects.

- `repo_scaffold/cli.py` is the CLI entry point. It loads the template registry, implements `repo-scaffold list`, delegates `repo-scaffold create <template>` to Cookiecutter, wires the `gh-init` command to the GitHub bootstrap layer, and provides the `add-package` command for adding packages to workspace projects.
- `repo_scaffold/github_init/` is a package with three modules: `__init__.py` orchestrates `init_repository`/`git_push`/`deploy_docs`, `config.py` holds `GhInitConfig`/`GhInitResult` and `build_config` (resolving defaults from `pyproject.toml`, `.env`, environment variables, and prompts), and `client.py` has `GhInitClient` (a thin PyGithub wrapper). After a successful push, unless `--no-pages`, `init_repository` creates the `gh-pages` branch and configures the repo's GitHub Pages source via the REST API (`GhInitClient.ensure_branch`/`enable_pages`, since PyGithub 2.x exposes no Pages helper); with `--protect-branch`, it protects the default branch via `Branch.edit_protection` with `enforce_admins=False` so the release token can still push. Pages and protection failures are surfaced but never abort the bootstrap.
- `repo_scaffold/add_package/` is a package with three modules: `__init__.py` exposes `add_package()` (detect project type and delegate), `config.py` holds `ProjectType` enum (RUST_WORKSPACE / UV_WORKSPACE / PNPM_WORKSPACE), `AddPackageConfig` dataclass, and `detect_project_type()` (reads `Cargo.toml` for `[workspace]` → Rust, `pyproject.toml` for `[tool.uv.workspace]` → uv, or `pnpm-workspace.yaml` → pnpm), and `workspace.py` implements `add_rust_package()` (creates crate skeleton + appends `[packages.<name>]` to `cog.toml` with `cargo workspaces version` hook + `cargo check`), `add_uv_package()` (runs `uv init --lib` + appends to `cog.toml` with `uv version --package` hook + `uv sync`), and `add_pnpm_package()` (writes package.json/vite.config.ts/tsconfig.json/src/index.ts + appends to `cog.toml` with `pnpm --filter <name> version` hook + `pnpm install`). The template-embedded `scripts/add_package.py` files are deprecated in favor of this CLI command.
- `repo_scaffold/templates/cookiecutter.json` is the registry of discoverable templates. Add new templates there so both `repo-scaffold list` and `repo-scaffold create <template>` can find them.
- Template source lives under `repo_scaffold/templates/template-python/`, `repo_scaffold/templates/template-uv-workspace/`, `repo_scaffold/templates/template-react/`, `repo_scaffold/templates/template-rust/`, `repo_scaffold/templates/template-pnpm-workspace/`, `repo_scaffold/templates/template-ts-sdk/`, and `repo_scaffold/templates/template-vue-project/`. Files below each `{{cookiecutter.project_slug}}/` are copied into generated projects and may contain Jinja expressions.
- Each template has its own `cookiecutter.json` defining prompts/defaults and a `hooks/post_gen_project.py` that validates rendered values, removes optional files, initializes a git repo on branch `master` (unless `create --no-git` sets `init_git=no`), and optionally runs dependency installation (`uv sync` for the single-package template, `uv sync --all-groups` for the workspace template, `pnpm install` for the React/pnpm-workspace/ts-sdk templates, `cargo build` for the Rust template).
- The single-package template can optionally include a Click CLI, Podman compose files, GitHub Actions, and MkDocs docs. The uv-workspace template generates a workspace root plus an initial package under `packages/` and can optionally include GitHub Actions/docs. The React template generates a TanStack Start (SSR React) project with TanStack Router/Query/Form/Store, MUI, Tailwind CSS, Biome for lint/format, pnpm for deps, Vitest for testing, and optional Docker/Podman support, GitHub Actions CI, and demo pages. The Rust template generates a cargo workspace with Axum + SQLx (PostgreSQL), domain-driven architecture (domain/infra/api/dto layers), JWT auth middleware, and optional Docker/Podman, GitHub Actions CI, OpenAPI/Swagger (utoipa), and OpenTelemetry support. The ts-sdk template generates a TypeScript SDK library with Vite lib mode (dual ESM+CJS), Prettier, and optional GitHub Actions (CI + dual npm/GPR publish + Cocogitto version bump). The pnpm-workspace template generates a pnpm monorepo root with an initial sub-package chosen from four variants (vue-app / ts-lib / react-app / ts-cli). The vue-project template generates a standalone Vue 3 project with Vue Router, Pinia, and Tailwind CSS, featuring a full layered component structure (components/ui, components/layout, components/common, pages/ with colocated sub-components, router/, stores/).

## Template-specific notes

- The top-level project and generated templates all use `justfile` for developer workflows; keep shared recipe behavior in sync only when the same behavior is intended in both places.
- The root `.ruff.toml` and coverage config exclude `repo_scaffold/templates/**/*`, so generated-template Python/Jinja files are not linted or measured by the root Ruff/coverage settings.
- Be careful editing files inside `repo_scaffold/templates/*/{{cookiecutter.project_slug}}/`: preserve Cookiecutter/Jinja delimiters and raw blocks, especially in GitHub Actions YAML and any literal `${{ ... }}` or `{{ ... }}` syntax.
- Optional template files are removed by `post_gen_project.py` after rendering; do not rely on conditional directory omission unless you also update the hook and tests.
- The React template uses `biome.json` (not `.ruff.toml`) for lint/format config and `pnpm` (not `uv`) for dependency management. Its justfile wraps `pnpm` commands and has conditional Docker/Podman recipes wrapped in Jinja `{% if %}` blocks.
- The Rust template uses `cargo clippy`/`cargo fmt` for lint/format and `cargo` for dependency management. Its workspace root `Cargo.toml` uses `[workspace.dependencies]` for shared versions. OpenAPI and OpenTelemetry are controlled via Jinja conditionals in source files (e.g., `utoipa` annotations, `#[cfg(feature = "opentelemetry")]`), while Docker and GitHub Actions are whole-file conditionals handled by `post_gen_project.py`.
- The ts-sdk template uses Prettier for formatting and pnpm for dependency management. It builds in Vite lib mode producing dual ESM+CJS output with `vite-plugin-dts` for type declarations. Path aliases (`@*` → `src/*`) are configured in both `vite.config.ts` and `Btsconfig.json`. `axios` and `tslog` are peerDependencies externalized from the build.
- The pnpm-workspace template uses Prettier for formatting and pnpm for dependency management. Sub-package variants are stored as `_vue-app/`, `_ts-lib/`, `_react-app/`, `_ts-cli/` directories under `packages/`; the `post_gen_project.py` hook renames the chosen variant to `{{cookiecutter.package_slug}}/` and removes the others. Cocogitto version bump uses `pnpm --filter <name> version` hooks.
- The vue-project template generates a standalone Vue 3 app with Vue Router, Pinia, and Tailwind CSS v4 (`@tailwindcss/vite`). It uses Prettier and pnpm. All `.vue` files use the Composition API with `<script setup>` and TypeScript; Vue template syntax (`{{ ... }}`, `v-for`, `v-if`, `:props`) is wrapped in `{% raw %}...{% endraw %}` blocks to avoid Jinja conflicts. `{{cookiecutter.project_slug}}` variables used inside `{% raw %}` blocks must be extracted into a `<script setup>` const first (see `AppHeader.vue`'s `projectName`).
- **Shared workflow fragments**: The pnpm templates (react, ts-sdk, pnpm-workspace, vue-project) deduplicate CI/version-bump step sequences via Jinja `{% include %}`. Fragment files live under `{{cookiecutter.project_slug}}/_shared/steps/` (e.g. `pnpm-setup.yaml` for the checkout→pnpm→node→install sequence, `pnpm-env.yaml` for react's version-bump which skips the install). Workflows reference them with `{% include '_shared/steps/<file>.yaml' %}` — the fragment is inlined at 6-space indentation during rendering. Cookiecutter also renders the `_shared/` files as standalone output, so each pnpm template's `post_gen_project.py` calls `clean_shared_fragments()` to remove the `_shared/` directory after rendering. To change a shared step across all pnpm templates, edit the fragment in each template's `_shared/steps/` directory (one edit per template, not per workflow).
- **Renovate dependency updates**: Every template includes a `.github/renovate.json5` that configures [Renovate](https://docs.renovatebot.com/) for automated dependency updates in generated projects. The config is controlled by `use_github_actions` — when `no`, the entire `.github/` directory (including Renovate config) is removed by the post-gen hook. Each template uses a preset and manager set appropriate to its ecosystem:
  - Python/uv-workspace: `config:best-practices` + `pep621` + `github-actions` managers; dev deps automerge.
  - React: `config:best-practices` + `npm` + `github-actions`; TanStack and MUI packages grouped; dev deps automerge.
  - ts-sdk: `config:js-lib` + `npm` + `github-actions`; preserves semver ranges for prod/peer deps (library); dev deps automerge.
  - pnpm-workspace: `config:best-practices` + `npm` + `github-actions`; dev deps automerge.
  - vue-project: `config:js-app` + `npm` + `github-actions`; pins all except peer deps (app); dev deps automerge.
  - Rust: `config:best-practices` + `cargo` + `github-actions`; minor/patch cargo deps automerge.
  - All templates automerge minor/patch GitHub Actions updates. Major updates require manual review.
  - The repo-scaffold project itself also has `.github/renovate.json5` (pep621 + github-actions).

## Tests and validation

- `tests/test_import.py` imports all non-template modules under `repo_scaffold`.
- `tests/test_templates.py` validates the template registry, CLI template resolution, Cookiecutter rendering for the Python and uv-workspace templates, static rendered-project checks, and post-generation hook behavior. The React and Rust templates are not yet covered by automated tests.
- `tests/test_github_init.py` unit-tests dotenv parsing, config resolution, PyGithub wrapper behavior, CLI wiring, and the git push helper for `gh-init`.
- `tests/test_add_package.py` unit-tests project-type detection (Cargo.toml `[workspace]` vs pyproject.toml `[tool.uv.workspace]`), Rust/uv workspace add-package logic, cog.toml section generation, and CLI wiring for `add-package`.
- For CLI/template changes, supplement `just test` with a manual generation check such as `uv run repo-scaffold create python --no-input --no-install -o <temp-dir>` because most tests validate static rendering, not full generated-project workflows after dependency installation. For React and Rust template changes this is especially important since no automated tests exist yet: `uv run repo-scaffold create react --no-input --no-install -o <temp-dir>` or `uv run repo-scaffold create rust --no-input --no-install -o <temp-dir>`.
- CI runs `uvx --from rust-just just init`, `uvx --from rust-just just lint`, `uvx --from rust-just just lint-pre-commit`, and `uvx --from rust-just just test-version 3.12` on pushes and pull requests.

## Release and docs

- Versioning is managed by [Cocogitto](https://docs.cocogitto.io/) (`cog.toml`); `pre_bump_hooks` run `uv version {{version}}` and `uv lock --no-upgrade`, and `post_bump_hooks` push commits/tags. Tags use plain semver like `0.15.1`.
- The version bump workflow (`.github/workflows/version-bump.yaml`) uses `cocogitto/cocogitto-action@v3` with `release: true` to compute the next version from conventional commits, write `CHANGELOG.md`, commit, and tag. The release workflow then publishes tagged builds.
- The release workflow publishes tagged builds to the private PyPI server by default and conditionally to public PyPI when `PUBLISH_TO_PUBLIC_PYPI` is true.
- MkDocs uses generated pages from `docs/gen_ref_pages.py` and `docs/gen_home_pages.py`; API reference generation excludes templates and the home page is generated from `README.md`.
